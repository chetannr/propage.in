Architecture Verdict GitHub Pages serves visitors directly via its built-in Fastly CDN, with a Let's Encrypt cert that GitHub provisions and auto-renews on its own schedule. Cloudflare hosts DNS only — gray cloud, permanently. No proxy means no cert renewal trap, no API tokens, no scheduled workflows. The right tradeoff when traffic is modest (under ~10,000 visits/month), audience is concentrated in regions GitHub's Fastly CDN already serves well (US, EU), and operational simplicity matters more than edge caching. For higher-traffic or globally-distributed audiences, see Plan B.
Target architecture

Simpler. One TLS hop. One renewing cert. No middle layer.

Visitors connect directly to GitHub Pages over HTTPS. GitHub serves the site through its built-in Fastly CDN with a Let's Encrypt certificate it provisions and renews on its own. Cloudflare's job is DNS only — it answers the lookup with GitHub's IPs and steps out of the request path entirely.

02
GitHub Pages · Custom Deploy

Commit the deploy workflow and enable Pages via GitHub Actions

Modern GitHub Pages deployments run via a custom GitHub Actions workflow rather than the legacy "deploy from branch" mode. The workflow below builds on every push to main, refreshes sitemap.xml's <lastmod> date automatically for SEO, and publishes through GitHub's official deploy-pages action. This is the same deploy.yml used by Plan B — one file, identical between plans, validated in production.

Add .github/workflows/deploy.yml:

name: Deploy static site to GitHub Pages

on:
  push:
    branches: [ main ]
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Update sitemap lastmod for deploy
        run: |
          if [ -f sitemap.xml ]; then
            ISODATE=$(date +%Y-%m-%d)
            sed -i "s/<lastmod>.*<\/lastmod>/<lastmod>${ISODATE}<\/lastmod>/g" sitemap.xml
          fi

      - name: Setup Pages
        uses: actions/configure-pages@v5

      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: '.'

      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4

Then in the GitHub UI:

  1. Repo → Settings → Pages.
  2. Under Build and deployment → Source, select GitHub Actions (not "Deploy from a branch"). No other settings on this page need changing — the workflow handles everything.
  3. Push any commit to main, or trigger manually via Actions tab → "Deploy static site to GitHub Pages" → Run workflow. The workflow completes in ~30–60 seconds.
  4. Confirm the site loads at the URL shown in the Pages settings panel — https://{GITHUB_USER}.github.io for user/org sites, or https://{GITHUB_USER}.github.io/{REPO} for project sites — before touching DNS.
✓ sitemap.xml auto-refreshes on every deploy

If you maintain a sitemap.xml at the repo root, the workflow rewrites every <lastmod> tag to today's date during the deploy step. Search engines pick up the freshness signal without manual editing — faster re-crawls, better SEO posture.

✓ Event-driven, not recurring

Unlike Plan B's cert-renewal, heartbeat, and cache-purge workflows (which run on schedules), deploy.yml only fires when you push to main. There's no cron job to keep alive, no scheduled task that GitHub can auto-disable. It's a single workflow file that just sits there until a push triggers it.

03
Cloudflare · DNS

Configure DNS records (DNS-only, permanently)

In Cloudflare → zone {DOMAIN} → DNS → Records, the records below must be the only A / AAAA / CNAME records on @ and www. Each one's Proxy status must be DNS only (gray cloud) — and stay that way forever. Unlike Plan B, there's no later step that flips them to proxied.

⚠ Why DNS-only is permanent here

With the proxy on, GitHub's DNS check sees Cloudflare's edge IPs instead of 185.199.x.153 and rejects cert provisioning. Even if you get past that initially, ACME-based cert renewal at the 90-day mark fails because Let's Encrypt can't reach origin. Plan A sidesteps this entirely by keeping the proxy off. The trade-off: no edge caching, no DDoS shield, no HTTP/3.

⚠ Pre-flight: audit existing Cloudflare Rules

Before adding any DNS records, navigate to Cloudflare → Rules → Overview and check three panels:

  • Page Rules (being phased out, but still active if present)
  • Redirect Rules (the modern replacement)
  • Configuration Rules

Delete anything that targets @, www, or the apex/subdomain combination unless you intentionally want to keep it. Legacy rules from a prior setup are invisible from the DNS panel and will silently override your new configuration — the classic symptom is ERR_TOO_MANY_REDIRECTS on apex or www after DNS propagates.

1. Clean up first. Delete any existing A / AAAA / CNAME records on @ or www that would conflict.

2. Add the apex A records — four of them, all DNS only:

A   @   185.199.108.153   DNS only
A   @   185.199.109.153   DNS only
A   @   185.199.110.153   DNS only
A   @   185.199.111.153   DNS only

3. Optionally add IPv6 — four AAAA records, also DNS only:

AAAA  @  2606:50c0:8000::153  DNS only
AAAA  @  2606:50c0:8001::153  DNS only
AAAA  @  2606:50c0:8002::153  DNS only
AAAA  @  2606:50c0:8003::153  DNS only

4. Add the www CNAME — single record, DNS only:

CNAME   www   {GITHUB_USER}.github.io   DNS only

5. Verify from the terminal:

dig +short {DOMAIN}
dig +short www.{DOMAIN}

The apex should return exactly the four 185.199.x.153 IPs. If it returns Cloudflare IPs (104.x.x.x or 172.64.x.x), a record is still proxied — flip it to DNS only before continuing.

04
GitHub · Cert + HTTPS

Bind the domain, wait for the cert, enforce HTTPS

  1. Repo → Settings → Pages → Custom domain: enter {DOMAIN} and click Save. GitHub runs a DNS check — should turn green within seconds since the proxy is off.
  2. GitHub provisions a TLS cert via Let's Encrypt. Usually a few minutes; the docs allow up to 24 hours. The Enforce HTTPS checkbox stays disabled with a status message until ready.
  3. Once the checkbox activates, tick Enforce HTTPS. GitHub now redirects all http:// requests to https://, and redirects whichever of apex or www is not set as the custom domain to the one that is.
✓ Auto-renewal handled by GitHub

Every ~60 days, GitHub silently renews the Let's Encrypt cert via the same ACME flow. Because the proxy is off, the renewal just works — no quarterly cert dance, no workflows to manage, no failure self-reporting needed. You'll never think about cert expiry again.

05
Verification

Verify and you're done

From the terminal:

curl -vI https://{DOMAIN} 2>&1 | grep -iE "issuer|subject"

Look for:

  • Issuer: Let's Encrypt (specifically R3, R10, R11, or a similar Let's Encrypt intermediate)
  • Subject: {DOMAIN}

In a browser, click the padlock and confirm the issuer is Let's Encrypt — not Cloudflare. There is no Cloudflare cert in this architecture; if the browser shows one, something is misconfigured.

Finally, push a small content change to test the deploy cycle: git push → wait ~30–60 seconds for GitHub's Pages build → reload. Change should appear globally via Fastly.

✓ Pass

Cert issuer is Let's Encrypt, content updates within ~60 seconds of push. You're live.

✗ Fail — issuer is Cloudflare

A DNS record is proxied. Go back to Step 3 and confirm every A / AAAA / CNAME on @ and www shows the gray cloud (DNS only) in Cloudflare's DNS panel.

⚠ Fail — DNS check stays red in GitHub

Either DNS hasn't propagated yet (wait 5–30 minutes) or a record is proxied (returns Cloudflare IPs to GitHub's check). Re-verify with dig +short {DOMAIN} — if it returns 104.x.x.x or 172.64.x.x, that's Cloudflare's edge; you need 185.199.x.153.

Common pitfalls

  • Orange cloud on ANY record → GitHub's DNS check sees Cloudflare's edge IPs instead of 185.199.x.153 and rejects cert provisioning. Enforce HTTPS stays disabled. In Plan A, the gray cloud is permanent — there is no later step that flips it.
  • Legacy Page Rules / Redirect Rules → pre-existing Cloudflare rules from a prior setup survive across DNS changes, cert reissuance, and proxy flips. They're invisible from the DNS panel where this runbook spends most of its time, but they still execute. Symptom: ERR_TOO_MANY_REDIRECTS on apex or www despite a correct DNS + cert + Enforce HTTPS setup. Fix: Rules → Overview, audit Page Rules + Redirect Rules + Configuration Rules, delete anything targeting apex or www unless intentional. The Step 3 pre-flight callout catches this before it bites.
  • Missing CNAME file → GitHub may overwrite the custom domain setting on the next deploy. Always commit a CNAME file at the repo root with {DOMAIN} as the sole content.
  • Lowercase cname filename → GitHub Pages expects the file to be named CNAME exactly (uppercase, no extension). Case matters on Linux runners.
  • Apex CNAME flattening → never put a CNAME on @. Use the four A records. Cloudflare supports flattening but GitHub's docs and verification expect A records.
  • Private repo without paid plan → Pages on private repos requires GitHub Pro/Team/Enterprise. Free accounts must use a public repo.
  • Bandwidth ceiling → GitHub Pages has a soft cap of 100 GB/month. If you start approaching it (track via GitHub's billing dashboard), it's time to migrate to Plan B for Cloudflare's edge caching.
