Repository

  1. Push the static site to a repo: {GITHUB_USER}/{REPO}.
  2. The repo must be public unless the org is on GitHub Pro/Team/Enterprise — Pages on private repos requires a paid plan.
  3. At the root of the repo, create a file named CNAME (uppercase, no extension) with exactly:

02
GitHub Pages · Custom Deploy

Commit the deploy workflow and enable Pages via GitHub Actions

Modern GitHub Pages deployments run via a custom GitHub Actions workflow rather than the legacy "deploy from branch" mode. The workflow below builds on every push to main, refreshes sitemap.xml's <lastmod> date automatically for SEO, and publishes through GitHub's official deploy-pages action.

Add .github/workflows/deploy.yml:

name: Deploy static site to GitHub Pages

on:
  push:
    branches: [ main ]
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Update sitemap lastmod for deploy
        run: |
          if [ -f sitemap.xml ]; then
            ISODATE=$(date +%Y-%m-%d)
            sed -i "s/<lastmod>.*<\/lastmod>/<lastmod>${ISODATE}<\/lastmod>/g" sitemap.xml
          fi

      - name: Setup Pages
        uses: actions/configure-pages@v5

      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: '.'

      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4

Then in the GitHub UI:

  1. Repo → Settings → Pages.
  2. Under Build and deployment → Source, select GitHub Actions (not "Deploy from a branch"). No other settings on this page need changing — the workflow handles everything.
  3. Push any commit to main, or trigger manually via Actions tab → "Deploy static site to GitHub Pages" → Run workflow. The workflow completes in ~30–60 seconds.
  4. Confirm the site loads at the URL shown in the Pages settings panel — https://{GITHUB_USER}.github.io for user/org sites, or https://{GITHUB_USER}.github.io/{REPO} for project sites — before touching DNS.
✓ sitemap.xml auto-refreshes on every deploy

If you maintain a sitemap.xml at the repo root, the workflow rewrites every <lastmod> tag to today's date during the deploy step. Search engines pick up the freshness signal without manual editing — faster re-crawls, better SEO posture.

⚠ Step 7a chains off this workflow's name:

The cache-purge workflow in Step 7a fires when this deploy workflow completes, matching by name: field. If you rename this workflow from "Deploy static site to GitHub Pages" to anything else, you must update the workflows: array in cloudflare-purge.yml to match — otherwise the cache won't auto-purge on deploys and stale content will linger for up to 30 days.

03
Cloudflare · DNS

Configure DNS records (temporarily DNS-only)

In Cloudflare → zone {DOMAIN} → DNS → Records, the records below must end up being the only A / AAAA / CNAME records on @ and www. Each one's Proxy status must be DNS only (gray cloud) for now — Step 5 flips them to Proxied once GitHub has issued its Let's Encrypt cert.

⚠ Why DNS-only first

GitHub's DNS check verifies the apex resolves directly to GitHub Pages' IPs (185.199.108–111.153). With Cloudflare's proxy on, the apex resolves to Cloudflare's edge IPs (104.x.x.x) instead — GitHub's check fails and cert issuance never starts. Step 5 turns the proxy on intentionally once the cert is in place.

1. Clean up first. Delete any existing A / AAAA / CNAME records on @ or www that would conflict with what's added below. Doing this before adding new records avoids a window where stale and new records coexist.

2. Add the apex A records — four of them, all on @, all DNS only:

A   @   185.199.108.153   DNS only
A   @   185.199.109.153   DNS only
A   @   185.199.110.153   DNS only
A   @   185.199.111.153   DNS only

3. Optionally add IPv6 — four AAAA records on @, also DNS only:

AAAA  @  2606:50c0:8000::153  DNS only
AAAA  @  2606:50c0:8001::153  DNS only
AAAA  @  2606:50c0:8002::153  DNS only
AAAA  @  2606:50c0:8003::153  DNS only

4. Add the www CNAME — single record, DNS only:

CNAME   www   {GITHUB_USER}.github.io   DNS only

5. Verify from the terminal before moving on:

dig +short {DOMAIN}
dig +short www.{DOMAIN}

The apex should return exactly the four 185.199.x.153 IPs. If it returns Cloudflare IPs (104.x.x.x or 172.64.x.x), at least one record is still proxied — flip it back to DNS only before continuing.

04
GitHub · Cert + HTTPS

Bind the domain, wait for the cert, enforce HTTPS

  1. Repo → Settings → Pages → Custom domain: enter {DOMAIN} and click Save. GitHub runs a DNS check — should turn green within seconds since the proxy is off.
  2. GitHub provisions a TLS cert via Let's Encrypt. Usually a few minutes; the docs allow up to 24 hours. The Enforce HTTPS checkbox stays disabled with a status message until ready.
  3. Once the checkbox activates, tick Enforce HTTPS. GitHub now redirects all http:// requests to https://, and redirects whichever of apex or www is not set as the custom domain to the one that is (server-side, at GitHub).
  4. Quick verify from the terminal: 
    curl -vI https://{DOMAIN} 2>&1 | grep -iE "issuer|subject"
    Issuer should be Let's Encrypt at this stage. After Step 5 it will become Cloudflare Inc — that's the goal. (Using grep -iE for portability across GNU and BSD grep.)
05
Cloudflare · Proxy + TLS

Activate the Cloudflare proxy

Cert exists, HTTPS is enforced. Promote Cloudflare from passthrough into the proper edge:

  1. SSL/TLS → Overview: set encryption mode to Full (strict). This must happen before flipping the proxy — Cloudflare needs to verify GitHub's Let's Encrypt cert on every origin pull, and Full (strict) is what enforces that.
  2. DNS → Records: flip every record from DNS only (gray) to Proxied (orange) — all four A records, IPv6 if added, and the www CNAME.
  3. SSL/TLS → Edge Certificates: enable the edge HTTPS features:
    • Always Use HTTPS: ON (Cloudflare-side redirect, faster than GitHub's)
    • Automatic HTTPS Rewrites: ON
    • Minimum TLS Version: TLS 1.2
    • TLS 1.3: ON
    • Opportunistic Encryption: ON
✗ Never use Flexible SSL

Flexible presents HTTPS to users but talks HTTP to the origin. GitHub's Enforce HTTPS then redirects every request, Cloudflare follows the redirect — infinite loop.

✓ GitHub Enforce HTTPS stays ON

GitHub's Enforce HTTPS works cleanly with Cloudflare in Full (strict). Both sides use HTTPS end-to-end — no redirect loops.

✓ The 90-day cert renewal is fully automated

GitHub's Let's Encrypt cert at origin expires every 90 days and would normally fail to renew once the proxy is on (Let's Encrypt can't reach origin directly through Cloudflare). Step 7b handles this with a GitHub Action that runs every 6 hours, detects the cert nearing expiry, briefly flips records to DNS only so GitHub can renew, then restores the proxy automatically. If anything ever fails, Step 7b self-reports by opening a GitHub issue. Step 7c's heartbeat ensures the renewal workflow itself never gets auto-disabled by GitHub. Zero manual intervention, indefinitely.

06
Speed · Network · Cache

Enable caching and performance

Speed → Optimization:

  • Brotli compression: ON (better than gzip for text assets — Cloudflare applies this at the edge)
  • Early Hints: ON (sends 103 Early Hints with preload directives)
  • Rocket Loader: leave OFF — rewrites JS and frequently breaks modern frameworks and third-party scripts
⚠ Auto Minify is gone

Cloudflare deprecated Auto Minify on August 5, 2024. The toggles for HTML/CSS/JS minification no longer appear in the dashboard. Minify at the build step instead — every modern static-site framework (Astro, Hugo, Eleventy, Next.js export, Vite, etc.) does this by default. For raw HTML/CSS/JS sites, add html-minifier-terser, esbuild, or similar to the build script.

Network:

  • HTTP/3 (QUIC): ON
  • 0-RTT Connection Resumption: ON
  • IPv6 Compatibility: ON

Caching → Configuration:

  • Caching Level: Standard
  • Browser Cache TTL: 4 hours (origin Cache-Control still takes precedence)
  • Always Online: ON — serves a Wayback Machine snapshot when GitHub Pages has an outage
  • Tiered Cache: ON — free, consolidates origin pulls across Cloudflare regions
  • Crawler Hints: ON
  • Development Mode: OFF unless actively debugging

Caching → Cache Rules — add one rule to cache the whole static site aggressively:

Rule name:        Cache static site
When matching:    Hostname equals {DOMAIN}
Cache eligibility: Eligible for cache
Edge TTL:          Override origin → 1 month
Browser TTL:       Respect origin   (or override to 4 hours)

Steady state: first request to a URL → cf-cache-status: MISS, Cloudflare fetches from GitHub. Every subsequent request for the next 30 days → cf-cache-status: HIT served from the edge with zero GitHub Pages bandwidth.

07
CI · Automation Workflows

Automate cache purge and cert renewal

Two GitHub Actions workflows go in .github/workflows/. Both run on the free tier, both fire automatically, both eliminate the manual maintenance burden of running a cached static site behind Cloudflare.

7a · Cache purge on every deploy

Every push triggers Step 2's deploy.yml workflow, but Cloudflare keeps serving the cached version until edge TTL expires (we set 30 days). Without automated purge, content changes won't show up for a month. This workflow chains off deploy.yml by name and purges automatically on every successful publish:

Add .github/workflows/cloudflare-purge.yml:

name: Purge Cloudflare cache
on:
  workflow_run:
    workflows: ["Deploy static site to GitHub Pages"]
    types: [completed]
jobs:
  purge:
    if: ${{ github.event.workflow_run.conclusion == 'success' }}
    runs-on: ubuntu-latest
    steps:
      - name: Purge Everything
        run: |
          curl -X POST "https://api.cloudflare.com/client/v4/zones/${{ secrets.CF_ZONE_ID }}/purge_cache" \
            -H "Authorization: Bearer ${{ secrets.CF_API_TOKEN }}" \
            -H "Content-Type: application/json" \
            --data '{"purge_everything":true}'

Add two repository secrets (Repo → Settings → Secrets and variables → Actions):

  • CF_ZONE_ID — Cloudflare → zone overview, right sidebar
  • CF_API_TOKEN — Cloudflare → My Profile → API Tokens → Create Token. Permissions: Zone · Cache Purge · Purge. Zone Resources: Include → Specific zone → {DOMAIN} (least-privilege, single zone).

7b · Auto-renew GitHub's origin cert at 90 days

GitHub's Let's Encrypt cert at origin expires every 90 days and renews via ACME HTTP-01 — which fails once Cloudflare's proxy is on. Without this workflow, the cert dies at day 90 and Cloudflare's Full (strict) mode starts returning 526. The workflow below runs every 6 hours, detects impending expiry, briefly flips records to DNS only so GitHub can renew, then restores the proxy and purges cache.

Add .github/workflows/cert-renewal.yml:

name: Origin cert auto-renewal

on:
  schedule:
    - cron: '0 */6 * * *'   # every 6 hours
  workflow_dispatch:

permissions:
  pages: write
  contents: read
  issues: write      # for the failure self-report step

concurrency:
  group: cert-renewal
  cancel-in-progress: false

env:
  DOMAIN: ${{ vars.CUSTOM_DOMAIN }}
  ZONE_ID: ${{ secrets.CF_ZONE_ID }}
  CF_DNS_TOKEN: ${{ secrets.CF_API_TOKEN_DNS }}
  CF_PURGE_TOKEN: ${{ secrets.CF_API_TOKEN }}

jobs:
  manage:
    runs-on: ubuntu-latest
    timeout-minutes: 15
    steps:
      - name: Inspect DNS state
        id: dns
        run: |
          curl -fsS \
            "https://api.cloudflare.com/client/v4/zones/$ZONE_ID/dns_records?per_page=100" \
            -H "Authorization: Bearer $CF_DNS_TOKEN" > records.json

          jq -r --arg dom "$DOMAIN" '.result[]
            | select((.name == $dom or .name == ("www." + $dom))
                    and (.type == "A" or .type == "AAAA" or .type == "CNAME"))
            | "\(.id) \(.proxied)"' records.json > records_state.txt

          if grep -q " false$" records_state.txt; then
            echo "proxy=off" >> "$GITHUB_OUTPUT"
          else
            echo "proxy=on" >> "$GITHUB_OUTPUT"
          fi
          cut -d' ' -f1 records_state.txt > record_ids.txt
          cat records_state.txt

      - name: Inspect origin cert expiry
        id: cert
        run: |
          # Probe a GitHub Pages IP directly with our domain's SNI.
          END_DATE=$(echo | timeout 10 openssl s_client \
            -connect 185.199.108.153:443 \
            -servername "$DOMAIN" 2>/dev/null \
            | openssl x509 -noout -enddate 2>/dev/null \
            | cut -d= -f2 || echo "")

          if [ -z "$END_DATE" ]; then
            echo "::warning::Cert probe failed; treating as expiring"
            DAYS=0
          else
            DAYS=$(( ($(date -d "$END_DATE" +%s) - $(date +%s)) / 86400 ))
            echo "::notice::Cert expires $END_DATE ($DAYS days)"
          fi
          echo "days=$DAYS" >> "$GITHUB_OUTPUT"

      - name: Decide action
        id: decide
        env:
          PROXY: ${{ steps.dns.outputs.proxy }}
          DAYS: ${{ steps.cert.outputs.days }}
        run: |
          if [ "$PROXY" = "on" ] && [ "$DAYS" -lt 25 ]; then
            echo "action=start" >> "$GITHUB_OUTPUT"
          elif [ "$PROXY" = "off" ] && [ "$DAYS" -gt 60 ]; then
            echo "action=finalize" >> "$GITHUB_OUTPUT"
          elif [ "$PROXY" = "off" ]; then
            echo "action=wait" >> "$GITHUB_OUTPUT"
          else
            echo "action=idle" >> "$GITHUB_OUTPUT"
          fi

      - name: Start - flip records to DNS only
        if: steps.decide.outputs.action == 'start'
        run: |
          while read ID; do
            curl -fsS -X PATCH \
              "https://api.cloudflare.com/client/v4/zones/$ZONE_ID/dns_records/$ID" \
              -H "Authorization: Bearer $CF_DNS_TOKEN" \
              -H "Content-Type: application/json" \
              --data '{"proxied":false}' > /dev/null
          done < record_ids.txt

      - name: Start - nudge GitHub Pages renewal
        if: steps.decide.outputs.action == 'start'
        run: |
          curl -fsS -X PUT \
            "https://api.github.com/repos/${{ github.repository }}/pages" \
            -H "Authorization: Bearer ${{ secrets.GITHUB_TOKEN }}" \
            -H "Accept: application/vnd.github+json" \
            --data "{\"cname\":\"$DOMAIN\",\"https_enforced\":true}" || \
              echo "::warning::Pages API call failed; GitHub will renew on its own cron"

      - name: Finalize - flip records back to Proxied
        if: steps.decide.outputs.action == 'finalize'
        run: |
          while read ID; do
            curl -fsS -X PATCH \
              "https://api.cloudflare.com/client/v4/zones/$ZONE_ID/dns_records/$ID" \
              -H "Authorization: Bearer $CF_DNS_TOKEN" \
              -H "Content-Type: application/json" \
              --data '{"proxied":true}' > /dev/null
          done < record_ids.txt

      - name: Finalize - purge cache
        if: steps.decide.outputs.action == 'finalize'
        run: |
          sleep 15
          curl -fsS -X POST \
            "https://api.cloudflare.com/client/v4/zones/$ZONE_ID/purge_cache" \
            -H "Authorization: Bearer $CF_PURGE_TOKEN" \
            -H "Content-Type: application/json" \
            --data '{"purge_everything":true}' > /dev/null

      - name: Self-report on failure
        if: failure()
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          RUN_URL="${{ github.server_url }}/${{ github.repository }}/actions/runs/${{ github.run_id }}"
          TS=$(date -u '+%Y-%m-%dT%H:%M:%SZ')

          # Look for existing open issue with our failure label
          EXISTING=$(curl -fsS \
            -H "Authorization: Bearer $GH_TOKEN" \
            -H "Accept: application/vnd.github+json" \
            "https://api.github.com/repos/${{ github.repository }}/issues?state=open&labels=cert-renewal-failure" \
            | jq -r '.[0].number // empty')

          if [ -n "$EXISTING" ]; then
            # Append a comment to the existing failure issue
            curl -fsS -X POST \
              -H "Authorization: Bearer $GH_TOKEN" \
              -H "Accept: application/vnd.github+json" \
              -H "Content-Type: application/json" \
              "https://api.github.com/repos/${{ github.repository }}/issues/$EXISTING/comments" \
              --data "$(jq -n --arg ts "$TS" --arg url "$RUN_URL" \
                '{body: ("Another failure on \($ts).\n\nRun: \($url)")}')" > /dev/null
          else
            # Create a new failure issue
            curl -fsS -X POST \
              -H "Authorization: Bearer $GH_TOKEN" \
              -H "Accept: application/vnd.github+json" \
              -H "Content-Type: application/json" \
              "https://api.github.com/repos/${{ github.repository }}/issues" \
              --data "$(jq -n --arg ts "$TS" --arg url "$RUN_URL" \
                '{title: "🚨 Cert auto-renewal workflow failed",
                  body: ("Workflow failed at \($ts). Run: \($url)"),
                  labels: ["cert-renewal-failure"]}')" > /dev/null
          fi

Additional setup beyond 7a:

  • Repo variable (not secret) CUSTOM_DOMAIN — Settings → Secrets and variables → Actions → Variables tab: set to your apex domain (e.g. example.com)
  • New repo secret CF_API_TOKEN_DNS — a second Cloudflare token, separate from CF_API_TOKEN:
    • Cloudflare → My Profile → API Tokens → Create Token
    • Permissions: Zone · DNS · Edit
    • Zone Resources: Include → Specific zone → {DOMAIN}

State machine in plain English:

  • Proxy on, cert fresh (>25d) → idle
  • Proxy on, cert expiring (<25d) → start: flip records to DNS-only, nudge Pages API
  • Proxy off, cert still old (<60d) → wait: GitHub renewing in background
  • Proxy off, cert renewed (>60d) → finalize: flip back to Proxied, purge cache
✓ Idempotent and crash-safe

State lives in Cloudflare itself (proxy on/off). The workflow is stateless — safe to retry, safe to run manually, and self-recovers if any single run fails. The job uses concurrency to prevent overlapping runs from racing each other.

⚠ Brief CDN window during renewal

While the proxy is off (typically 1–12 hours per renewal cycle, ~4 times per year), the site loses Cloudflare's caching, DDoS protection, and edge HTTPS. Visitors still get served correctly — just from GitHub directly. For high-traffic or high-stakes sites where this isn't acceptable, migrate the origin to Cloudflare Pages instead.

7c · Heartbeat — keep the renewal workflow alive

GitHub auto-disables scheduled workflows after 60 days of repo inactivity to save compute on dormant projects. For a marketing site with regular content updates this never triggers, but for a near-dormant repo, the cert-renewal workflow in 7b would silently stop running — and at day 90 the cert would expire with no one watching. This heartbeat workflow updates a single file in .github/ twice a month. Each update is a real commit, which resets GitHub's 60-day inactivity timer. The system stays self-sustaining indefinitely.

Add .github/workflows/heartbeat.yml:

name: Heartbeat — keep scheduled workflows active

on:
  schedule:
    - cron: '0 0 1,15 * *'   # 1st and 15th of every month at 00:00 UTC
  workflow_dispatch:

permissions:
  contents: write

jobs:
  beat:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Update heartbeat file
        run: |
          mkdir -p .github
          {
            echo "# Heartbeat"
            echo ""
            echo "Last beat: $(date -u '+%Y-%m-%dT%H:%M:%SZ')"
            echo ""
            echo "Updated automatically twice a month by .github/workflows/heartbeat.yml"
            echo "to prevent GitHub from auto-disabling scheduled workflows. Do not edit."
          } > .github/HEARTBEAT.md

      - name: Commit if changed
        run: |
          git config user.name "github-actions[bot]"
          git config user.email "41898282+github-actions[bot]@users.noreply.github.com"
          git add .github/HEARTBEAT.md
          if git diff --cached --quiet; then
            echo "::notice::No changes to commit"
          else
            git commit -m "chore: heartbeat $(date -u '+%Y-%m-%d')"
            git push
          fi

Setup: none. The workflow uses only the default GITHUB_TOKEN with contents: write permission — no secrets to add, no tokens to manage.

✓ Self-sustaining for life

The heartbeat workflow's own commits count as repo activity, which keeps all scheduled workflows (including itself) eligible to run. The system needs no external life support; it just runs.

08
Verification

Verify the live, cached, auto-purging site

From the terminal:

curl -I https://{DOMAIN}
curl -I https://{DOMAIN}    # second hit

Look for:

  • server: cloudflare — confirms traffic is proxied
  • cf-cache-status: MISS on first request → HIT on subsequent
  • cf-ray: <id> — Cloudflare request ID

In a browser, click the padlock and check the certificate issuer. Should be Cloudflare Inc — the edge cert. Let's Encrypt is still doing its job behind the scenes, just no longer visitor-facing.

Finally, push a small content change to test the full cycle: git push → Pages builds → purge workflow fires → reload. The change should appear globally within ~90–120 seconds (see the Deploy Timeline below).

✓ Pass

All headers present, cert issuer is Cloudflare Inc, content updates within 2 minutes of push.

✗ Fail — issuer is still Let's Encrypt

The records aren't proxied. Go back to Step 5 and confirm every record shows orange cloud in the Cloudflare DNS panel.

⚠ cf-cache-status: DYNAMIC

Cloudflare decided not to cache. Re-check the Cache Rule from Step 6 is enabled and matches the hostname exactly.

──────── STEP 5 — Activate the Cloudflare proxy ────────
Critical: do these sub-steps in EXACT order. Wrong order = broken TLS chain or redirect loop.

1. Navigate to Cloudflare → SSL/TLS → Overview.
2. Confirm the current encryption mode. If it is anything other than "Full (strict)", change it to "Full (strict)". Read aloud what it was before and what it is now.

3. Navigate to Cloudflare → DNS → Records.
4. For each record on @ and www (the four A records, IPv6 if added, and the www CNAME), flip the proxy toggle from gray cloud (DNS only) to orange cloud (Proxied). Verify each visually before continuing.
5. List the final records back to me with proxy status. Every record should be orange cloud.
6. Navigate to SSL/TLS → Edge Certificates. Turn ON:
   • Always Use HTTPS
   • Automatic HTTPS Rewrites
   • Opportunistic Encryption
   Set Minimum TLS Version: TLS 1.2. Set TLS 1.3: Enabled.

Common pitfalls

  • Orange cloud during Steps 3 or 4 → GitHub's DNS check sees Cloudflare's edge IPs instead of 185.199.x.153 and fails. Cert provisioning never starts and Enforce HTTPS stays disabled. Records must be DNS only until Step 5 flips the proxy on intentionally.
  • Cloudflare Flexible SSL + GitHub redirects = infinite redirect loop. SSL mode must be Full (strict) by Step 5.
  • Flipping the proxy before Full (strict) → broken TLS chain. The SSL mode change in Step 5 must happen before the records go orange.
  • Forgetting the CNAME file in the repo → GitHub will overwrite the custom domain setting on the next deploy.
  • Missing CF_ZONE_ID or CF_API_TOKEN secrets → the cache-purge workflow fails silently on every deploy and stale content lingers.
  • Missing automation workflows → the cert renewal (7b) and heartbeat (7c) workflows must both be deployed. Without 7b, the cert expires at day 90 and Cloudflare returns 526. Without 7c, GitHub auto-disables 7b after 60 days of repo inactivity and you're back to the same failure. Verify both workflow files are committed, all three secrets are set, the CUSTOM_DOMAIN variable is populated, and that both workflows have been manually triggered once each via the Actions tab to confirm they succeed.
  • Apex CNAME flattening → don't put a CNAME on @. Use the four A records. Cloudflare supports flattening but GitHub's docs and verification expect A records.
