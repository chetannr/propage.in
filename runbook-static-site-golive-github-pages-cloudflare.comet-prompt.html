Static site go-live.
GitHub Pages,

The standard procedure every team must follow when launching a static website on GitHub Pages with DNS managed by Cloudflare. HTTPS is enforced by GitHub via a Let's Encrypt certificate — Cloudflare's SSL features stay off.

⌗ Variables

Substitute these in every command

Throughout this runbook, the highlighted placeholders below appear in code, file contents, and Cloudflare/GitHub configuration. Replace each with your project's actual value before executing.

{DOMAIN}
The apex domain you're publishing — e.g. example.com, acme-careers.io
{GITHUB_USER}
The GitHub user or organisation that owns the repo — e.g. acme-corp
{REPO}
The repository name holding the static site source — e.g. acme-marketing-site
Core Principle GitHub issues a Let's Encrypt cert for the custom domain, and that issuance only works if Cloudflare's proxy is off — gray cloud, DNS only. Anything proxied breaks the ACME challenge and disables the Enforce HTTPS checkbox.
01
Repository

Prepare the GitHub repo

  1. Push the static site to a repo: {GITHUB_USER}/{REPO}.
  2. The repo must be public unless the org is on GitHub Pro/Team/Enterprise — Pages on private repos requires a paid plan.
  3. At the root of the repo, create a file named CNAME (uppercase, no extension) with exactly: 
    {DOMAIN}
    Commit and push.
02
GitHub Pages

Enable GitHub Pages

  1. Go to Repo → Settings → Pages.
  2. Under Build and deployment → Source, choose either Deploy from a branch (select main and /root) or GitHub Actions if a workflow is used to build.
  3. Confirm the site loads at https://<{GITHUB_USER}>.github.io (or …/{REPO} for project sites) before touching DNS.
03
Cloudflare · DNS Records

Configure DNS in Cloudflare

In Cloudflare → zone {DOMAIN} → DNS → Records, create the following. Every record's Proxy status must be DNS only (gray cloud).

Apex ({DOMAIN}) — four A records:

A   @   185.199.108.153   DNS only
A   @   185.199.109.153   DNS only
A   @   185.199.110.153   DNS only
A   @   185.199.111.153   DNS only

Optional but recommended IPv6 — also DNS only:

AAAA  @  2606:50c0:8000::153  DNS only
AAAA  @  2606:50c0:8001::153  DNS only
AAAA  @  2606:50c0:8002::153  DNS only
AAAA  @  2606:50c0:8003::153  DNS only

www — single CNAME, also DNS only:

CNAME   www   {GITHUB_USER}.github.io   DNS only
⚠ Cleanup

Delete any old A / AAAA / CNAME records on @ or www that conflict with the above.

04
Cloudflare · SSL/TLS

Disable Cloudflare's SSL features

Since the proxy is off, most of these don't apply — but turn them off anyway so nothing interferes later.

  1. SSL/TLS → Overview: the mode is irrelevant when no record is proxied, but if you want to be safe set it to Full (strict). Never use Flexible with GitHub Pages.
  2. SSL/TLS → Edge Certificates: turn off Always Use HTTPS and Automatic HTTPS Rewrites. GitHub will handle these.
  3. Rules / Page Rules / Redirect Rules: remove any HTTP→HTTPS or domain-rewrite rules pointing at {DOMAIN}.
05
GitHub · Custom Domain

Bind the custom domain in GitHub

  1. Go to Repo → Settings → Pages → Custom domain, enter {DOMAIN}, and click Save.
  2. GitHub runs a DNS check. Wait for the green check on the apex and on www — it verifies the four A records and the CNAME.
  3. Resolution should be fast since the proxy is off; if it fails, recheck the records.
06
Certificate Provisioning

Wait for the Let's Encrypt certificate

  1. After the DNS check passes, GitHub provisions a TLS cert via Let's Encrypt.
  2. This usually takes a few minutes; the docs allow up to 24 hours.
  3. The Enforce HTTPS checkbox under Pages stays greyed out with a status message until the cert is issued.
07
HTTPS · Enforce

Enforce HTTPS

  1. Once the cert is ready, the checkbox becomes active.
  2. Go to Repo → Settings → Pages → Enforce HTTPS and enable it.
  3. GitHub will now redirect all http:// requests to https:// and serve www ↔ apex according to the CNAME.
08
Verification

Verify everything from the terminal

dig +short {DOMAIN}
dig +short www.{DOMAIN}
curl -I https://{DOMAIN}
curl -I http://{DOMAIN}    # expect 301 → https
curl -vI https://{DOMAIN} 2>&1 | grep -i "issuer\|subject"
✓ Pass

The cert issuer should be Let's Encrypt.

✕ Fail

If the issuer is Cloudflare Inc ECC CA, the proxy is still on somewhere — switch that record back to DNS only.

═══════════════════════════════════════════════
THE 8 STEPS
═══════════════════════════════════════════════

──────── STEP 1 — Verify the GitHub repo and CNAME file ────────
1. Open https://github.com/{GITHUB_USER}/{REPO}
2. Confirm the repo loads and is accessible.
3. Check whether a file named CNAME (uppercase, no extension) exists in the repo root.
   • If it exists and contains exactly "{DOMAIN}" → OK, report and continue.
   • If it exists but contains a different value → STOP and ask me what to do.
   • If it does not exist → create it via the GitHub web UI:
       - Click "Add file" → "Create new file"
       - Filename: CNAME
       - Content: {DOMAIN}    (no trailing whitespace, no quotes, no extra lines)
       - Commit directly to the default branch with message: "Add CNAME for {DOMAIN}"
4. Report the commit URL.
5. End of Step 1 — request approval.

──────── STEP 2 — Enable GitHub Pages ────────
1. Navigate to https://github.com/{GITHUB_USER}/{REPO}/settings/pages
2. Read aloud the current state of "Build and deployment → Source".
3. If Pages is not yet enabled:
   • Source: "Deploy from a branch"
   • Branch: "main"  /  Folder: "/ (root)"
   • Click Save.
   (If the user mentions they use GitHub Actions for build, ask before changing source.)
4. Wait until the page shows the live URL (https://{GITHUB_USER}.github.io/... or .../{REPO}). Report it.
5. Open that URL in a new tab and confirm the site loads.
6. End of Step 2 — request approval.

──────── STEP 3 — Configure DNS in Cloudflare ────────
1. Open https://dash.cloudflare.com and navigate to the {DOMAIN} zone → DNS → Records.
2. List every existing A, AAAA, and CNAME record on "@" (apex) and "www". Show them to me.
3. Ask me to confirm which conflicting records may be deleted. Wait for my "confirm".
4. After my confirmation, delete the conflicting records.
5. Add the following records, one at a time. For EACH record, set Proxy status = "DNS only" (gray cloud). Verify the proxy toggle visually before saving.

   Apex A records:
     A   @   185.199.108.153   DNS only
     A   @   185.199.109.153   DNS only
     A   @   185.199.110.153   DNS only
     A   @   185.199.111.153   DNS only

   Optional IPv6 (ask me yes/no before adding):
     AAAA  @  2606:50c0:8000::153  DNS only
     AAAA  @  2606:50c0:8001::153  DNS only
     AAAA  @  2606:50c0:8002::153  DNS only
     AAAA  @  2606:50c0:8003::153  DNS only

   www CNAME:
     CNAME   www   {GITHUB_USER}.github.io   DNS only

6. After each record is saved, re-read the records list and confirm the new row is present and gray-cloud. If any record shows orange cloud, fix it before continuing.
7. Show me a final summary table of all records on @ and www.
8. End of Step 3 — request approval.

──────── STEP 4 — Disable Cloudflare's SSL/HTTPS features ────────
1. Go to SSL/TLS → Overview. Read the current encryption mode aloud.
   • If it is "Flexible", change it to "Full (strict)" and tell me. Never leave it on Flexible.
   • Otherwise, leave it as-is.
2. Go to SSL/TLS → Edge Certificates.
   • Turn OFF "Always Use HTTPS"
   • Turn OFF "Automatic HTTPS Rewrites"
   • Confirm both toggles read OFF.
3. Go to Rules → Page Rules and Rules → Redirect Rules. List any rule whose target includes {DOMAIN}.
   • For each such rule, ask me to confirm before disabling/deleting it.
4. End of Step 4 — request approval.

──────── STEP 5 — Bind the custom domain in GitHub ────────
1. Return to https://github.com/{GITHUB_USER}/{REPO}/settings/pages
2. Under "Custom domain", enter: {DOMAIN}
3. Click Save.
4. Wait for GitHub's DNS check. Report the status messages exactly as they appear.
5. If the DNS check fails, capture the exact error text and STOP. Do not retry without my approval.
6. End of Step 5 — request approval.

──────── STEP 6 — Wait for the Let's Encrypt certificate ────────
1. On the same Pages settings screen, locate the "Enforce HTTPS" checkbox.
2. Report whether it is currently enabled, disabled, or grayed-out with a status message.
3. If it is grayed out with a "certificate is being provisioned" type message, INFORM me that this can take a few minutes up to 24 hours, and PAUSE. Do not poll repeatedly. Ask me to tell you when to re-check.
4. End of Step 6 — request approval (or pause).

──────── STEP 7 — Enforce HTTPS ────────
1. When I tell you to re-check, refresh the Pages settings screen.
2. If the "Enforce HTTPS" checkbox is now active, tick it.
3. Confirm the checkbox is now in the enabled state and report it.
4. End of Step 7 — request approval.

──────── STEP 8 — Verification ────────
1. Open a new tab and visit each of the following, reporting status code and final URL:
     • http://{DOMAIN}        (expect 301 → https)
     • https://{DOMAIN}
     • http://www.{DOMAIN}    (expect 301 → https)
     • https://www.{DOMAIN}
2. On https://{DOMAIN}, click the padlock and read the certificate issuer. Report it.
   • If the issuer is "Let's Encrypt" → PASS.
   • If the issuer is "Cloudflare Inc ECC CA" or similar → FAIL: a Cloudflare proxy is still on somewhere. Go back to Step 3, find the orange-cloud record, switch it to DNS only, and re-run Step 8.
3. Provide a final go-live report:
     ✓ DNS records (with proxy state)
     ✓ Custom domain bound
     ✓ Cert issuer
     ✓ Enforce HTTPS state
     ✓ All four URLs and their final status codes
4. End of Step 8 — go-live complete. Confirm I have no follow-up changes.

COMMON PITFALLS — WATCH FOR THESE THROUGHOUT
═══════════════════════════════════════════════
• Orange cloud on ANY record breaks GitHub's cert issuance. Always verify gray cloud after saving.
• Cloudflare "Flexible" SSL + GitHub redirects = infinite redirect loop. Never use Flexible.
• Don't put a CNAME on @ even if Cloudflare offers flattening — use the four A records.
• Don't delete the CNAME file in the repo; GitHub will overwrite the custom domain on next deploy if it's missing.
• Never share session cookies, tokens, or 2FA codes in chat.

Watch For

Common pitfalls

  • Orange cloud on any record → cert won't issue, Enforce HTTPS stays disabled, or the Cloudflare cert is served instead of GitHub's.
  • Cloudflare Flexible SSL + GitHub redirects = infinite redirect loop. Don't use Flexible.
  • Forgetting the CNAME file in the repo → GitHub will overwrite the custom domain setting on the next deploy.
  • Apex CNAME flattening → don't put a CNAME on @. Use the four A records. Cloudflare supports flattening but GitHub's docs and verification expect A records.
